# CTT Research Report: Context-Time Training across 11 LLMs

**RepoMemory + A2E Protocol — Empirical study**
**Date**: March 2026
**Models tested**: 11 (1B to 120B parameters)
**Infrastructure**: Cloudflare Workers AI
**Protocol**: A2E v1.0.0

---

## 1. Executive Summary

We tested whether RepoMemory's Context-Time Training (CTT) can enable any LLM — regardless of size, architecture, or training data — to generate valid A2E (Agent-to-Execution) workflows. None of the 11 models were trained on A2E. All learned it exclusively from context injected by RepoMemory.

**Key finding**: Models with >= 3B active parameters and >= 24K context window produce valid A2E workflows, most on the first attempt. Experience accumulated by one model transfers to any other model via RepoMemory, eliminating the need for per-model fine-tuning.

---

## 2. What is CTT (Context-Time Training)

CTT trains the **agent**, not the **model**. Instead of modifying weights via fine-tuning, CTT stores experience (successful patterns, corrections, knowledge) in RepoMemory and injects it as context at inference time.

### The CTT cycle

```
┌─────────────────────────────────────────────────────┐
│  1. RepoMemory recalls knowledge + experience       │
│  2. Context injected as system prompt                │
│  3. LLM generates A2E workflow                       │
│  4. Validator checks output                          │
│  5a. Valid → save as successful pattern (fact)       │
│  5b. Invalid → save as correction (2x boost)         │
│  6. Next call gets enriched context → improved output│
└─────────────────────────────────────────────────────┘
```

### Correction boost

Corrections are stored with `category: 'correction'` and receive a **2x scoring multiplier** in RepoMemory's recall engine. This ensures mistakes are surfaced first in future context, before the model can repeat them.

### Cross-model inheritance

Experience is model-agnostic. A correction generated by gemma-7b-it (e.g., "use JSONL format, not single JSON object") is inherited by llama-3.2-1b, qwq-32b, or any other model that queries the same RepoMemory instance.

## 3. Test Methodology

### A2E knowledge base

9 knowledge entries (8 primitives + JSONL format) and 7 skill entries (workflow examples) ingested via `ingestA2EKnowledge()`. Total: ~8,000 chars of A2E documentation stored in RepoMemory.

### CTT experience

5-7 correction memories accumulated from previous model runs, covering:
- Nested operation format (`{"operation":{"ApiCall":{...}}}`)
- JSONL format (one JSON per line)
- Required `beginExecution` message
- `/workflow/` path prefix
- JSON brace completion

### Test cases

| Test | Description | Primitives expected |
|------|-------------|-------------------|
| Simple | GET users + filter active | ApiCall, FilterData |
| Complex | 2 APIs + merge + filter + sort + save | ApiCall, MergeData, FilterData, TransformData, StoreData |
| POST | POST with JSON body | ApiCall (POST) |

### Validation

`validateWorkflow()` checks: valid JSON per line, valid message types, nested operation structure, primitive-specific field validation, cross-reference of operationIds, and beginExecution presence.

`fixJsonl()` auto-corrects: unquoted keys, unquoted string values, unquoted URLs/paths, trailing commas, single quotes.

## 5. Analysis: What Determines Success

### 5.1 Instruction-following capability (primary factor)

The dominant variable is not knowledge (RepoMemory provides that) but the model's ability to follow structured output instructions:

- **Produce multiple JSON lines** (not a single object, not pretty-printed)
- **Include all required parts** (operationUpdate + beginExecution)
- **Maintain internal consistency** (operationIds in operationOrder match defined ones)
- **Complete JSON correctly** (matching braces, no truncation)

Models >= 20B active params do this reliably on first attempt. Models at 1-7B do it with pipeline optimizations (auto-repair + auto-beginExecution). Only gemma-2b (2B + 8K context) fails — the dual constraint of minimal parameters AND minimal context is the true floor.

### 5.2 Context window (secondary factor)

| Context | Impact |
|---------|--------|
| >= 24K | Full A2E knowledge + corrections + few-shot examples fit comfortably |
| 8K | Must reduce context to ~3,500 chars. Fewer few-shot examples, fewer corrections. Works only if model has strong instruction-following |
| Unlimited (128K+) | No benefit over 24K for A2E tasks — the knowledge base is ~8K chars total |

Evidence: granite-h-micro (~3B, 131K context) outperformed gemma-7b-it (7B, 8K context) because it received the full knowledge base while gemma received a truncated version.

### 5.3 Model architecture (tertiary factor)

| Architecture | Impact on A2E |
|-------------|---------------|
| Dense (gpt-oss, llama, gemma, granite) | Performance scales linearly with params |
| MoE (nemotron 120B/12B active) | 12B active performs like >= 20B dense |
| Reasoning (qwq-32b) | Produces chain-of-thought before JSONL — works after stripping `<think>` tags |
| Code-specialized (qwen2.5-coder) | Excellent at structured output but occasionally omits non-code parts (beginExecution) |

### 5.4 Function calling capability (not required)

A2E explicitly does NOT require function calling / tool use capability:

| Model | Function calling | A2E result |
|-------|-----------------|------------|
| gpt-oss-120b | Yes (via API) | OK |
| nemotron-3-120b | Yes (native) | OK |
| qwq-32b | Yes (tools API) | OK |
| gpt-oss-20b | Yes | OK |
| granite-h-micro | No | OK |
| gemma-7b-it | No | OK (2 iter) |
| llama-3.2-3b | No | OK (5 iter) |
| llama-3.2-1b | No | Partial |

**Function calling is irrelevant to A2E**. The LLM generates a declarative workflow definition (JSONL text), not function calls. This is a fundamental architectural advantage of A2E over MCP/tools:

- **MCP/Tools**: requires the model to have function calling capability, generates one tool call at a time, accumulates context with each round-trip
- **A2E**: requires only text generation, defines the entire workflow in one shot, no round-trips

This means models like granite-h-micro (~3B, no function calling) can fully participate in A2E workflows — something impossible with MCP.

## 6. Error Taxonomy

### 6.1 Pipeline-auto-fixed errors (resolved without CTT iterations)

These errors are now handled transparently by the three-stage pipeline (`normalizeResponse` → `fixJsonl` → `validateWorkflow`):

| Error | Fix stage | Models affected |
|-------|-----------|----------------|
| JSON truncation (missing `}`) | `repairTruncatedJson` in fixJsonl | llama-3.2-3b, llama-3.2-1b |
| Missing beginExecution | `validateWorkflow` auto-synthesis | llama-3.2-3b, llama-3.2-1b, qwen-coder |
| Pretty-printed multi-line JSON | `normalizeResponse` collapse | gemma-2b, gemma-7b |
| `<think>` reasoning tags | `normalizeResponse` strip | qwq-32b |
| Markdown code blocks | `normalizeResponse` extract | gemma-7b, gemma-2b |
| beginExecution placed first | `normalizeResponse` reorder | gemma-2b |
| Unquoted keys/values | `fixJsonl` regex | gpt-oss-20b |
| Trailing commas | `fixJsonl` regex | Various |
| Single quotes | `fixJsonl` regex | Various |

### 6.2 CTT-correctable errors (resolved by adding corrections to RepoMemory)

| Error | Models affected | Iterations to fix |
|-------|----------------|-------------------|
| Single JSON object instead of JSONL | gemma-7b-it | 1-2 |
| Flat operation format | Early tests (all models) | 1 |
| Wrong operator string ("gt" vs ">") | llama-1b | 1 |
| Missing body in POST | llama-1b | 1-2 |

### 6.3 Non-fixable errors (model limitations)

| Error | Models affected | Root cause |
|-------|----------------|------------|
| Inconsistent operationIds | gemma-2b-it-lora | Defines `fetch-users` but references `get-data` — semantic incoherence |
| Extra characters in JSON (trailing commas inside values) | gemma-2b-it-lora | Malformed generation below repair threshold |
| Ignores query semantics | gemma-2b-it-lora | Copies examples verbatim instead of adapting to query |

### 6.4 The dual-constraint floor

The true floor is not a single variable but the combination of **small parameters AND small context window**. Models with 1B params but 60K context (llama-1b) now converge. Models with 2B params but only 8K context (gemma-2b) do not — because there isn't enough room for both the A2E knowledge and the corrections that would fix their output.

## 7. MoE vs Dense: The Nemotron Finding

| Model | Architecture | Total params | Active params | Simple | Complex | POST |
|-------|-------------|-------------|--------------|--------|---------|------|
| gpt-oss-120b | Dense | 120B | 120B | OK 1 | OK 1 | OK 1 |
| nemotron-3-120b | MoE | 120B | 12B | OK 1 | OK 1 | OK 1 |
| gpt-oss-20b | Dense | 20B | 20B | OK 1 | OK 1 | OK 1 |

Nemotron's 12B active parameters produce output **identical in quality** to gpt-oss-120b's 120B dense. This suggests that for structured output tasks like A2E:

- MoE's expert routing effectively selects relevant parameters
- 12B active MoE ≈ 20B+ dense for instruction-following tasks
- The cost efficiency is significant: MoE uses ~10% of the compute for equivalent results

**Implication for CTT**: MoE models are ideal CTT targets — they provide large-model quality at small-model cost, and CTT provides the domain knowledge they lack.

## 8. Token Efficiency: A2E vs MCP

Measured in previous tests with actual API calls:

| Scenario | MCP tokens | A2E tokens | Savings |
|----------|-----------|-----------|---------|
| Simple (1 API call) | ~800 | ~600 | 25% |
| Complex (5-step pipeline) | ~4,500 | ~500 | 89% |

The savings increase with workflow complexity because:
- **MCP**: each tool call requires a round-trip (send tool schema → model generates call → execute → return result → model processes → next call). Context grows linearly.
- **A2E**: the entire workflow is defined in one generation. No round-trips. Constant context.

For a 5-step workflow, MCP requires 5 round-trips accumulating ~900 tokens each. A2E requires 1 generation of ~500 tokens total.

## 9. Practical Recommendations

### For production deployments

| Priority | Model class | Recommendation |
|----------|------------|----------------|
| Quality first | >= 20B dense or MoE | Use gpt-oss-120b, nemotron, qwq-32b. Valid on first attempt, no pipeline fixes needed |
| Cost-optimized | 1-7B with >= 24K context | Use llama-3.2-1b, llama-3.2-3b, granite-h-micro. Pipeline auto-fixes handle truncation + missing beginExecution. Valid on first attempt post-optimization |
| Not recommended | <= 2B AND <= 8K context | gemma-2b-it-lora. Dual constraint (tiny params + tiny window) is below the floor. Use only with human-in-the-loop |

### For CTT knowledge design

1. **Always include few-shot examples** in skills with `[A2E: ...]` format — RecallEngine extracts these as user/assistant pairs
2. **Keep corrections specific and short** — long corrections compete for limited context in small-window models
3. **Use `saveOrUpdate` with source-based dedup** — prevents correction accumulation from bloating context
4. **Set `template: 'a2e'`** in recall — applies 1.5x memory weight and enables few-shot extraction

### For A2E workflow validation

1. **Use `validateWorkflow()` directly** — it runs the full pipeline: `normalizeResponse()` → `fixJsonl()` → validate + auto-fix. No need to call individual functions
2. **Check `result.autoFixed`** — if `true`, beginExecution was auto-synthesized (the model omitted it). Log this for CTT learning
3. **Handle object responses** — some models (qwen-coder) return parsed JSON objects instead of strings via Cloudflare API. Cast with `JSON.stringify()` before passing to `validateWorkflow()`
4. **Set `max_tokens >= 1000`** for complex workflows — small models need room to generate all lines

## 10. Conclusions

### CTT + pipeline optimizations democratize AI capability

The knowledge lives in RepoMemory, not in model weights. After pipeline optimizations, a **1B model produces valid A2E workflows on the first attempt** — the same result as a 120B model. The pipeline compensates for the model's formatting deficiencies (truncation, missing parts) while the model provides the semantic understanding (correct primitives, paths, field values).

### The floor dropped from ~3B to ~2B

Before optimizations, models below ~3B couldn't reliably converge. After adding `repairTruncatedJson` and auto-`beginExecution` synthesis, **llama-3.2-1b (1B) converges in 1 iteration for simple and complex workflows**. The only remaining failure is gemma-2b-it-lora, which has the dual constraint of 2B parameters AND 8K context.

### The real limit is the dual constraint: tiny params + tiny context

A model with 1B params but 60K context works. A model with 7B params but 8K context works (with 2 iterations). A model with 2B params AND 8K context does not. The floor is not a single threshold — it's the combination. Either enough parameters to follow instructions OR enough context to include corrective examples.

### Validation pipeline > CTT corrections for formatting errors

The key insight from optimizations: **don't try to teach the model to close braces — just close them programmatically**. CTT corrections are valuable for semantic errors (wrong primitive, wrong structure) but wasteful for mechanical errors (truncation, missing boilerplate). The pipeline handles mechanical errors; CTT handles semantic ones.

### Function calling is irrelevant

A2E's text-based workflow definition works with any model that can generate text — including 1B models without function calling support. This is fundamentally different from MCP/tools which require function calling capability.

### MoE is the sweet spot

Nemotron (120B/12B active) matches 120B dense quality at a fraction of the cost. For CTT + A2E deployments, MoE models offer the best cost-performance ratio.

### One template serves all model sizes

The unified `a2e` template with skills-first ordering, direct preamble, and 7 few-shot examples works equally well for 1B and 120B models. Clear instructions don't hurt large models — they only help small ones.
